1. 保护模式初始化

1.1. 概述

处理器在硬件重置之后处于式模式。在将处理器切换到保护模式之前，软件初始化代码必须将最少数量的保护模式数据结构和代码模块加载到内存中，以支持处理器在保护模式下的可靠运行。相关数据结构如下：

  • 一个IDT表
  • 一个GDT表
  • 一个TSS段
  • 一个LDT表（可选）
  • 如果使用分页机制，至少一个页目录表和一个页表
  • 一个或多个包含必要的中断和异常处理程序的代码模块

在将处理器切换到保护模式之前，软件初始化代码还必须初始化以下系统寄存器：

  • GDTR寄存器
  • IDTR寄存器（可选）。此寄存器也可以在切换到保护模式后，启用中断之前进行初始化。
  • 控制寄存器CR1到CR4
  • 内存类型范围寄存器MTRR（部分处理器）
    初始化这些数据结构、代码模块和系统寄存器后，可以通过设置控制寄存器CR0中的PE标志（位0）的值，将处理器切换到保护模式。

1.2. 系统数据结构

软件初始化期间加载到内存中的保护模式系统数据结构的内容在很大程度上取决于保护模式操作系统或执行程序将支持的内存管理类型：扁平、扁平分页、分段或分段分页。
为了实现无分页的平面内存模型，必须创建一个至少拥有一个代码段描述符和一个数据段描述符的GDT表。在第一个GDT条目中还需要一个空描述符。堆栈可以放在一个普通的读/写数据段中，因此不需要专用的堆栈描述符。带分页的平面内存模型还需要创建至少一个页目录和一个页表（除非所有页都是4MB，在这种情况下只需要一个页目录）。
在使用GDT之前，必须使用LGDT指令将GDT的基址和长度加载到GDTR寄存器中。
多段模型的操作系统可能需要额外的段，以及每个应用程序的段和LDT表。LDT表需要在GDT表中有特定描述符。一些操作系统根据需要分配新的段和LDT表。这为处理动态编程环境提供了最大的灵活性。然而，许多操作系统对所有任务使用一个LDT，并且为其提前分配GDT条目。嵌入式系统（如进程控制器）可能会为固定数量的应用程序预先分配固定数量的段和LDT。这将是构建实时系统软件环境的一种简单而有效的方法。

1.3. 中断和异常

软件初始化期间至少加载一个IDT表，其中包含处理器可能生成的所有异常的门描述符。如果使用中断门或陷阱门，则门描述符都可以指向相同的代码段，其中包含必要的异常处理程序。如果使用任务门，则每个用任务门调用的异常处理程序都需要一个TSS段和相应的代码、数据和任务段。
如果允许外部中断，则必须在IDT中为每个中断处理程序提供门描述符。
在使用IDT之前，必须使用LIDT指令将IDT的基址和长度加载到IDTR寄存器中。此操作通常在切换到保护模式后立即执行。

1.4. 分页机制

在启用分页机制之前，必须初始化以下数据结构和寄存器

  • 必须加载至少一个页目录表和页表到内存中。如果页目录表包含指向自身的目录条目（此时，页目录表和页表位于同一页中），或者仅使用4MB的页，则可以不用页表。
  • CR3寄存器中需要加载页目录表的基地址。
  • （可选）可以在GDT或LDT中为管理模式提供一组代码和数据描述符，并为用户模式提供另一组代码和数据描述符。

当分页机制初始化完成时，分页被启用，同时通过设置CR0寄存器的PG和PE标志将处理器切换到保护模式。必须在处理器切换到保护模式之后启用分页。

1.5. 多任务机制

如果不启用多任务机制并且不允许在特权级别之间进行更改，则无需将TSS加载到内存中或初始化任务寄存器。
如果启用多任务机制或者允许在特权级别之间进行更改，则初始化时必须加载至少一个TSS段和相应的TSS描述符（切换特权级别需要使用TSS段是因为指向0、1、2特权级别栈段的指针和这些栈的栈指针都存储在TSS段中）。创建TSS描述符时，不能将其标记为busy，只能由处理器在任务切换时将其标记为busy。与LDT的描述符一样，TSS描述符驻留在GDT中。
当处理器切换到保护模式后，可以使用LTR指令将TSS描述符的段选择器加载到任务寄存器中。此指令将TSS描述符标记为忙碌，但不执行任务切换。但是，处理器可以使用TSS定位特权级别0、1和2堆栈的指针。在软件在保护模式下执行第一个任务切换之前，必须加载TSS的段选择器，因为任务切换会将当前任务状态复制到TSS中。
在执行LTR指令之后，通过任务切换对任务寄存器执行进一步的操作。与其他段和LDT段一样，TSS段和TSS描述符可以预先分配，也可以根据需要分配。

2. IA-32e模式初始化

处理器在进行切换IA-32e模式初始化操作之前必须处于保护模式并且开启分页机制。同时IA-32e模式要求使用4级或者5级分页结构。
操作系统初始化IA-32e模式的操作过程

  • 在保护模式下，使用MOV CR0指令设置CR0.PG=0，关闭分页。
  • 设置CR4.PAE=1，开启PAE(physical-address extensions)。
  • 将顶层页结构物理基地址加载到CR3寄存器中
  • 设置IA32_EFER.LME=1开启IA-32e模式。
  • 设置CR0.PG=1，开启分页。之后处理器会自动设置A32_EFER.LMA=1。当执行MOV CR0指令开启分页机制时，其后续指令必须位于同一性地址映射的页面内（直至处理器进入IA-32e模式后，才可以使用非同一性地址映射的页面）。

在进入IA-32e模式之前，64位模式的分页数据结构必须被放置在物理地址空间的开头4GB中。因为用于初始化页目录基地址的MOV CR3指令必须在激活IA-32e模式之前在原有模式下执行（设置CR0.PG=1以启用分页），所以此指令会在保护模式下执行，只能写入CR3寄存器的低32位，只能寻址4GB的地址空间。在进入IA-32e模式之后，可以将页表移动到任意位置。

如果试图改变IA32_EFER.LME、CR0.PG和CR4.PAE等影响IA-32e模式开启的标志位，处理器会进行64位模式的一致性检测，以确保处理器不会进入未定义模式或不可预测的运行状态。如果一致性检测失败，处理器将会产生通用保护性异常（#GP）。以下境遇会导致一致性检测失败。

  • 当开启分页机制后，再试图使能或禁止IA-32e模式。
  • 当开启IA-32e模式后，试图在开启物理地址扩展（PAE）功能前使能分页机制。
  • 在激活IA-32e模式后，试图禁止物理地址扩展（PAE）。
  • 当CS段寄存器的L位被置位时，再试图激活IA-32e模式。
  • 如果TR寄存器加载的是16位TSS段结构。

2.1. 模式数据结构

当IA-32e模式激活后，系统各描述符表寄存器（GDTR、LDTR、IDTR、TR）依然沿用保护模式的描述符表。由于保护模式的描述符表基地址是32位，这使得它们均位于低4 GB线性地址空间内。既然已经开启IA-32e模式，那么系统各描述符表寄存器理应（必须）重新加载（借助LGDT、LLDT、LIDT和LTR指令）为IA-32e模式的64位描述符表。

2.2. 中断和异常

当软件激活IA-32e模式后，中断描述符表寄存器IDTR仍然使用保护模式的中断描述符表，那么在将IDTR寄存器更新为64位中断描述符表IDT前不要触发中断和异常，否则处理器会把32位兼容模式的中断门解释为64位中断门，从而导致不可预料的结果。使用CLI指令能够禁止可屏蔽硬件中断，而NMI不可屏蔽中断，必须借助外部硬件电路才可禁止。
