Linux用户与用户组管理

基础知识
  • Linux系统是一个多用户多任务的分时操作系统，任何一个要使用系统资源的用户，都必须首先向系统管理员申请一个账号，然后以这个账号的身份进入系统
  • 用户的账号一方面可以帮助系统管理员对使用系统的用户进行跟踪，并控制他们对系统资源的访问，另一方面也可以帮助用户组织文件，并为用户提供安全性保护
  • 每个用户账号都拥有一个唯一的用户名和各自的口令
  • 用户在登录时键入正确的用户名和口令后，就能够进入系统和自己的主目录
  • 任何需要使用操作系统的用户都需要一个系统账号，账号分为：**管理员账号和普通用户账号**
  • 操作系统中根据UID来判断账户，而不是用户名，id为0即为管理员，可以有多个id为0的账户
  • 系统新建账号时，会根据账号类型，自动分配递增账号的UID和GID（用户身份编码，组编号），也可以自行分配。通常情况下应保证UID和GID唯一且不重复
组的类别
  • Linux中，每个用户必须拥有一个主组。创建账号时，系统会自动创建一个同名组作为主组。用户必须属于一个且只有一个主组。用户可以属于0个或者多个附加组。主组又称为基本组，不可以把用户从基本组中删除，但可以更换基本组。
  • 除了基本组之外，用户所在的其他组都是附加组。用户可以从附加组中删除。用户的附加组可以和主组相同，附加组可以直接删除，可以新增和移除任意个所属用户。用户被删除不会影响附加组。
用户与组的配置文件
  • 用户分为以下三类
    • 管理员：root，具有使用系统所有权限的用户，UID=0。
    • 普通用户：使用系统的权限受限，UID=500-60000。
    • 系统用户：保障系统运行的用户，一般不提供密码登录系统，UID=1-499。
  • 用户组配置文件
    • /etc/group，组及其属性信息
    • 使用cat /etc/group查看此文件（好像只能使用这种方式查看）

      其中的每一行表示系统中的一个用户组
    • 含义解释

      每一行都可以分为四列
      第一列：群组名称
      第二列：群组密码，真正的组密码存储在/etc/gshadow文件中
      第三列：群组的ID，即为GID
      第四列：属于这个组的所有用户的列表，此组可以是主组或者附加组，此处只显示将此组作为附加组的用户

  • 用户组密码配置文件
    • /etc/gshadow，组密码及其相关属性
    • 使用cat /etc/gshadow查看此文件
    • 含义解释

      每一行可以分为四列
      第一列：组名
      第二列：组密码
      第三列：组管理员
      第四列：属于这个组的所有用户的列表，此组可以是主组或者附加组，此处只显示将此组作为附加组的用户

  • 用户密码配置文件
    • /etc/shadow，用户密码及其相关属性
    • 其中的文件是和/etc/passwd中的文件一一对应的
    • 使用cat /etc/shadow查看此文件，只有超级用户才能读取此文件（为了保障密码的安全）
    • 含义解释

      每一行可以分为九列
      第一列：登录用户名
      第二列：密码，通常使用shad512加密，带有！表示该账号不能用来登录
      第三列：从1970年1月1日起计算到现在为止最近一次修改密码的时间
      第四列：密码再过几天可以被修改，0表示随时可以修改
      第五列：密码几天后必须变更，99999表示永不过期
      第六列：密码过期前多久提示用户
      第七列：密码过期多久后账户被锁定（用户没有登录活动但是账号仍然能够保持有效的最长时间），如果为0，密码过期直接失效，不能恢复
      第八列：多少天后账户将失效，（从1970-1-1开始算起）
      第九列：保留条目，暂未使用

  • 用户配置文件
    • /etc/passwd，用户及其属性信息
    • 使用cat /etc/passwd进行查看，任何用户都可以进行查看
    • 含义解释

      每一行可以分为七列
      第一列：登录用户名
      第二列：用户密码，真正的密码存储在/etc/gpasswd中
      第三列：用户ID，UId
      第四列：所属组的组ID,GID
      第五列：备注信息
      第六列：用户的主目录（中），当用户登录到到系统时就是在他的home中。是用户登录到系统之后所处的目录，主目录的名称即为用户的登录名,每个用户对自己的主目录有读写和执行的权限。
      第七列：shell类型。在用户登录后，需要启动一个进程，负责将用户的操作传递给内核，这个进程是用户登录到系统之后运行的命令解释器或者 某个特定的程序，即Shell，默认为/bin/sh

    • 伪用户
      • 在/etc/passwd中占有一条记录但是但是不能用于登录，因为他们的登录shell为空
      • 是为了方便系统管理，满足相应系统的进程对文件属主的要求
      • 常见的伪用户定义

        bin：拥有可执行的用户命令文件
        sys：拥有系统文件
        adm：拥有账户文件
        uucp：UUCP使用
        lp：lp或者lpd子系统使用
        nobody：NFS使用

Linux系统用户账号的管理
  • useradd
    • 添加新的用户账号,就是在/etc/passwd中为新用户增加一条信息，同时更新其他相关系统文件
    • 语法：useradd 选项 用户名
    • 参数说明
      • **-c**：指定一段注释性描述 -c comment
      • **-d**：指定用户主目录，如果不存在，同时使用-m选项，创建主目录 -d 目录 -m
      • **-g**：指定用户所属的基本组 -g 用户组
      • **-G**：指定用户所属的附加组 -G 用户组，用户组
      • **-s**：指定用户的登录shell -s shell文件
      • **-u**：指定用户的用户号，如果同时有-o选项，则可以重复使用其他用户的标识号 -u 用户号 -o
    • 各种参数可以配合使用，使用效果是单个效果的叠加。
    • 实例

      useradd -d /home/sam -m sam，添加一个以/home/sam为home的用户账号，账号名称为sam

      useradd -s /bin/sh -g group -G adm,root gem，添加一个gem用户，shell文件为/bin/sh，属于group主组，adm和root附加组
      （组如果不存在则应该先创建组）

  • userdel
    • 删除用户账号，就是将/etc/passwd等系统文件中的该用户信息删除，必要时还需要删除用户的主目录
    • 语法：userdel 选项 用户名
    • 参数说明
      • **-r**：将用户的主目录一起删除
    • 实例

      userdel -r sam，将之前创建的sam用户连同其主目录一同删除

  • usermod
    • 修改账号
    • 语法：usermod 选项 用户名
    • 参数说明
      • 同useradd
      • **-l**：指定一个新的账号，将原来的账号名更改为新的账号名，使用格式：-l 新用户名
      • **-p**：修改账号的口令，使用格式：-p 新口令
    • 实例

      usermod -d /home/sam -m sam，将sam账号的主目录进行更改

  • id
    • 查询账号口令
    • 语法：id 选项 用户名
    • 参数说明
      • **-u**：显示UID
      • **-g**：显示GID
      • **-G**：显示用户所属的组ID
      • **-n**：显示名称
    • 实例
  • passwd
    • 指定和修改用户口令
    • 用户刚刚创建时没有口令，但是被系统锁定，无法使用（在passwd中显示为!，表示无法登陆），必须为其指定口令之后才能使用，即使是空口令
    • 语法：passwd 选项 用户名
    • 参数说明
      • **-l**：锁定口令，禁用账号
      • **-u**：口令解锁
      • **-d**：使账号无口令，即为将口令删除，下一次将无法使用此账号登陆
      • **-f**：强迫用户下次登陆时修改口令
    • 省略用户名则默认为当前用户，超级用户可以修改任意用户的口令，普通用户只能修改自己的口令
    • 直接使用passwd可以修该当前用户的口令，使用passwd 用户名可以在root用户中修改任意账号的口令
Linux系统用户组的管理
  • groupadd
    • 添加新的用户组，用户组的操作实质上是对/etc/group等相关文件的修改
    • 语法：groupadd 选项 用户组
    • 参数说明
      • **-g**：指定新用户的组标识号（GID），默认情况下为递增，使用格式：-g GID
      • **-o**：一般与-g一同使用，表示新用户组的GID可以与系统已有用户组的GID相同
    • 实例

      groupadd a，添加新的用户组a，GID递增
      groupadd -g 1001 -o b，添加新的用户组b，GID为1001

  • groupdel
    • 删除已有的用户组
    • 语法：groupdel 用户组
  • groupmod
    • 修改用户组的属性
    • 语法：groupmod 选项 用户组
    • 参数说明
      • **-g**：为用户指定新的组标识号GID，使用格式：-g GID
      • **-o**：与-g同时使用，新GID可以与已有的GID相同
      • **-n**：将用户组的名字改为新名字，使用格式：-n 新名字
    • 实例
  • newgrp
    • 如果一个用户同时属于多个用户组，那么该用户可以在用户组之间进行切换，以便具有其他用户组的权限
    • 语法：newgrp 用户组
    • 注意：切换到的用户组必须是当前用户的主组或者附加组
  • groupmems
    • 用于更改和查看组中的成员
    • 语法：groupmems 选项 动作
    • 参数说明
      • 动作
      • **-g**：更改为指定组，只有root用户可以使用,使用格式：-g 用户组
      • 选项
      • **-a**：指定用户加入组，使用格式：-a 用户
      • **-d**：从组中删除用户，使用格式：-d 用户
      • **-p**：从组中清除所有成员
      • **-l**：显示组成员列表
    • 实例 显示adm组所有的成员
  • 添加批量用户
    • 用于快速创建大量用户
    • 具体步骤
      • 编辑一个文本用户文件，按照/etc/passwd中的格式书写，注意每个用户的用户名，UID，宿主目录都不相同，密码栏可以留空或者输入*


        此时系统会自动生成用户的密码(/etc/shadow)
      • 以root身份执行/usr/sbin/newusers，从刚创建的文件中导入数据，添加用户
      • 以root身份执行/use/sbin/pwunconv，此命令将/etc/shadow中的密码解码，写回到/etc/passwd中，并且将/etc/shadow的shadow密码栏删掉
      • 按照格式用户名 密码编辑用户的密码对照文件
      • 以root身份运行usr/sbin/chpasswd，此指令用于创建用户密码，会将经过/usr/sbin/passwd命令编码过的密码写入/etc/passwd的密码栏
      • 执行命令/usr/sbin/pwconv将密码编码为shadow passwd，并将结果写入/etc/shadow

状态管理

Cookie
  • 临时用来保存服务器端数据的技术，并且保存在浏览器端，敏感信息不建议保存在cookie中
  • 每个应用程序都对应有cookie，在cookie保存有当前cookie的路径，这个路径用于确定当前的cookie属于哪个应用程序
  • 访问请求时浏览器会自动判断当前的cookie路径是否和当前要访问的服务器一致，如果一致则将当前的数据打包放在请求数据包中一同发送给后台服务器。
Session
  • 用于临时保存服务器端数据的技术，并且将session保存在服务器端，可以用于保存(username,uid,avator)这些信息
  • 每一个session自动生成一个session id来唯一的标识当前的session对象。服务器响应数据时，会自动的将session id发送到浏览器端，浏览器接收当前的session id，将其存放到cookie当中。下一次访问服务器时，cookie会被自动提交给服务器（session id）。tomcat
  • 可设置存活事件，用于访问超时的实现。
  • session对象是一个全局对象，在项目中任何位置都可以获取到这个对象。

final关键字

  • 修饰类，则此类不能被继承
  • 修饰属性，则此变量的值不能被修改，称为常量
  • 修饰方法，则此方法不能被重写（方法名一致、参数列表一致、返回值类型一致、访问修饰符大于等于父类中被重写的方法）

1. 模式切换

1.1. 从实模式切换到保护模式

在从实模式切换到保护模式之前必须将相应的数据结构和代码模块加载到内存。执行MOV汇编指令置位CR0控制寄存器的PE标志位即可进入保护模式（可同时置位CR0寄存器的PG标志位以开启分页机制）。进入保护模式后，处理器将从0特权级（CPL=0）开始执行。
英特尔64和IA-32处理器对切换到保护模式的要求略有不同。为确保与Intel 64和IA-32处理器上下代码兼容，最好遵循以下步骤：

  • 关闭中断。执行CLI汇编指令禁止可屏蔽硬件中断，对于不可屏蔽中断NMI只能借助外部电路才能禁止。（模式切换程序必须保证在切换过程中不能产生异常和中断。）
  • 执行LGDT汇编指令将GDT的基地址和长度加载到GDTR寄存器。
  • 执行MOV CR0汇编指令位置CR0控制寄存器的PE标志位。（可同时置位CR0控制寄存器的PG标志位。）
  • 一旦MOV CR0汇编指令执行结束，紧随其后必须执行一条远跳转（far JMP）或远调用（farCALL）指令，以切换到保护模式的代码段去执行。（这是一个典型的保护模式切换方法。）
  • 通过执行JMP或CALL指令，可改变处理器的执行流水线，进而使处理器加载执行保护模式的代码段。
  • 如果开启分页机制，那么MOV CR0指令和JMP/CALL（跳转/调用）指令必须位于同一性地址映射的页面内。（因为保护模式和分页机制使能后的物理地址，与执行JMP/CALL指令前的线性地址相同。）至于JMP或CALL指令的目标地址，则无需进行同一性地址映射（线性地址与物理地址重合）。
  • 如需使用LDT，则必须借助LLDT汇编指令将GDT内的LDT段选择器加载到LDTR寄存器中。
  • 执行LTR汇编指令将一个TSS段描述符的段选择子加载到TR任务寄存器。处理器对TSS段结构无特殊要求，凡是可写的内存空间均可。
  • 进入保护模式后，数据段寄存器仍旧保留着实模式的段数据，必须重新加载数据段选择器或使用JMP/CALL指令执行新任务，便可将其更新为保护模式。（执行之前步骤的JMP或CALL指令已将代码段寄存器更新为保护模式。）对于不使用的数据段寄存器（DS和SS寄存器除外），可将NULL段选择子加载到其中。
  • 执行LIDT指令，将保护模式下的IDT表的基地址和长度加载到IDTR寄存器。
  • 执行STI指令使能可屏蔽硬件中断，并执行必要的硬件操作使能NMI不可屏蔽中断。

1. 保护模式初始化

1.1. 概述

处理器在硬件重置之后处于式模式。在将处理器切换到保护模式之前，软件初始化代码必须将最少数量的保护模式数据结构和代码模块加载到内存中，以支持处理器在保护模式下的可靠运行。相关数据结构如下：

  • 一个IDT表
  • 一个GDT表
  • 一个TSS段
  • 一个LDT表（可选）
  • 如果使用分页机制，至少一个页目录表和一个页表
  • 一个或多个包含必要的中断和异常处理程序的代码模块

在将处理器切换到保护模式之前，软件初始化代码还必须初始化以下系统寄存器：

  • GDTR寄存器
  • IDTR寄存器（可选）。此寄存器也可以在切换到保护模式后，启用中断之前进行初始化。
  • 控制寄存器CR1到CR4
  • 内存类型范围寄存器MTRR（部分处理器）
    初始化这些数据结构、代码模块和系统寄存器后，可以通过设置控制寄存器CR0中的PE标志（位0）的值，将处理器切换到保护模式。

1.2. 系统数据结构

软件初始化期间加载到内存中的保护模式系统数据结构的内容在很大程度上取决于保护模式操作系统或执行程序将支持的内存管理类型：扁平、扁平分页、分段或分段分页。
为了实现无分页的平面内存模型，必须创建一个至少拥有一个代码段描述符和一个数据段描述符的GDT表。在第一个GDT条目中还需要一个空描述符。堆栈可以放在一个普通的读/写数据段中，因此不需要专用的堆栈描述符。带分页的平面内存模型还需要创建至少一个页目录和一个页表（除非所有页都是4MB，在这种情况下只需要一个页目录）。
在使用GDT之前，必须使用LGDT指令将GDT的基址和长度加载到GDTR寄存器中。
多段模型的操作系统可能需要额外的段，以及每个应用程序的段和LDT表。LDT表需要在GDT表中有特定描述符。一些操作系统根据需要分配新的段和LDT表。这为处理动态编程环境提供了最大的灵活性。然而，许多操作系统对所有任务使用一个LDT，并且为其提前分配GDT条目。嵌入式系统（如进程控制器）可能会为固定数量的应用程序预先分配固定数量的段和LDT。这将是构建实时系统软件环境的一种简单而有效的方法。

1.3. 中断和异常

软件初始化期间至少加载一个IDT表，其中包含处理器可能生成的所有异常的门描述符。如果使用中断门或陷阱门，则门描述符都可以指向相同的代码段，其中包含必要的异常处理程序。如果使用任务门，则每个用任务门调用的异常处理程序都需要一个TSS段和相应的代码、数据和任务段。
如果允许外部中断，则必须在IDT中为每个中断处理程序提供门描述符。
在使用IDT之前，必须使用LIDT指令将IDT的基址和长度加载到IDTR寄存器中。此操作通常在切换到保护模式后立即执行。

1.4. 分页机制

在启用分页机制之前，必须初始化以下数据结构和寄存器

  • 必须加载至少一个页目录表和页表到内存中。如果页目录表包含指向自身的目录条目（此时，页目录表和页表位于同一页中），或者仅使用4MB的页，则可以不用页表。
  • CR3寄存器中需要加载页目录表的基地址。
  • （可选）可以在GDT或LDT中为管理模式提供一组代码和数据描述符，并为用户模式提供另一组代码和数据描述符。

当分页机制初始化完成时，分页被启用，同时通过设置CR0寄存器的PG和PE标志将处理器切换到保护模式。必须在处理器切换到保护模式之后启用分页。

1.5. 多任务机制

如果不启用多任务机制并且不允许在特权级别之间进行更改，则无需将TSS加载到内存中或初始化任务寄存器。
如果启用多任务机制或者允许在特权级别之间进行更改，则初始化时必须加载至少一个TSS段和相应的TSS描述符（切换特权级别需要使用TSS段是因为指向0、1、2特权级别栈段的指针和这些栈的栈指针都存储在TSS段中）。创建TSS描述符时，不能将其标记为busy，只能由处理器在任务切换时将其标记为busy。与LDT的描述符一样，TSS描述符驻留在GDT中。
当处理器切换到保护模式后，可以使用LTR指令将TSS描述符的段选择器加载到任务寄存器中。此指令将TSS描述符标记为忙碌，但不执行任务切换。但是，处理器可以使用TSS定位特权级别0、1和2堆栈的指针。在软件在保护模式下执行第一个任务切换之前，必须加载TSS的段选择器，因为任务切换会将当前任务状态复制到TSS中。
在执行LTR指令之后，通过任务切换对任务寄存器执行进一步的操作。与其他段和LDT段一样，TSS段和TSS描述符可以预先分配，也可以根据需要分配。

2. IA-32e模式初始化

处理器在进行切换IA-32e模式初始化操作之前必须处于保护模式并且开启分页机制。同时IA-32e模式要求使用4级或者5级分页结构。
操作系统初始化IA-32e模式的操作过程

  • 在保护模式下，使用MOV CR0指令设置CR0.PG=0，关闭分页。
  • 设置CR4.PAE=1，开启PAE(physical-address extensions)。
  • 将顶层页结构物理基地址加载到CR3寄存器中
  • 设置IA32_EFER.LME=1开启IA-32e模式。
  • 设置CR0.PG=1，开启分页。之后处理器会自动设置A32_EFER.LMA=1。当执行MOV CR0指令开启分页机制时，其后续指令必须位于同一性地址映射的页面内（直至处理器进入IA-32e模式后，才可以使用非同一性地址映射的页面）。

在进入IA-32e模式之前，64位模式的分页数据结构必须被放置在物理地址空间的开头4GB中。因为用于初始化页目录基地址的MOV CR3指令必须在激活IA-32e模式之前在原有模式下执行（设置CR0.PG=1以启用分页），所以此指令会在保护模式下执行，只能写入CR3寄存器的低32位，只能寻址4GB的地址空间。在进入IA-32e模式之后，可以将页表移动到任意位置。

如果试图改变IA32_EFER.LME、CR0.PG和CR4.PAE等影响IA-32e模式开启的标志位，处理器会进行64位模式的一致性检测，以确保处理器不会进入未定义模式或不可预测的运行状态。如果一致性检测失败，处理器将会产生通用保护性异常（#GP）。以下境遇会导致一致性检测失败。

  • 当开启分页机制后，再试图使能或禁止IA-32e模式。
  • 当开启IA-32e模式后，试图在开启物理地址扩展（PAE）功能前使能分页机制。
  • 在激活IA-32e模式后，试图禁止物理地址扩展（PAE）。
  • 当CS段寄存器的L位被置位时，再试图激活IA-32e模式。
  • 如果TR寄存器加载的是16位TSS段结构。

2.1. 模式数据结构

当IA-32e模式激活后，系统各描述符表寄存器（GDTR、LDTR、IDTR、TR）依然沿用保护模式的描述符表。由于保护模式的描述符表基地址是32位，这使得它们均位于低4 GB线性地址空间内。既然已经开启IA-32e模式，那么系统各描述符表寄存器理应（必须）重新加载（借助LGDT、LLDT、LIDT和LTR指令）为IA-32e模式的64位描述符表。

2.2. 中断和异常

当软件激活IA-32e模式后，中断描述符表寄存器IDTR仍然使用保护模式的中断描述符表，那么在将IDTR寄存器更新为64位中断描述符表IDT前不要触发中断和异常，否则处理器会把32位兼容模式的中断门解释为64位中断门，从而导致不可预料的结果。使用CLI指令能够禁止可屏蔽硬件中断，而NMI不可屏蔽中断，必须借助外部硬件电路才可禁止。

1. 01.分页模式

1.1. 概述

1. 逻辑地址、线性地址和物理地址

1.1. 物理地址

物理地址空间是指处理器可以通过其地址线访问的内存地址空间，在保护模式下，处理器提供了4GB(32)的物理地址空间。物理地址空间是不分段的，可以由任意类型的物理内存组成。

处理器还支持将物理内存扩展到64GB(36)。这种扩展可以使用以下方式开启

  • 使用CR4寄存器第5位，PAE标志位
  • 使用PSE-36属性

物理地址的支持已经扩展到36位以上。

1.2. 逻辑地址和线性地址

  • 作用：保护模式下处理器使用这两种地址转换得到物理地址（逻辑地址转换和线性地址分页）。
  • 逻辑地址
    • 处理器中的地址访问使用逻辑地址
    • 逻辑地址包含16位的段选择器和32位的偏移。段选择器选择数据所在的段，偏移指出数据在段中的偏移量。
    • 处理器会将逻辑地址翻译成线性地址
  • 线性地址
    • 线性地址是32位的地址，并且不分段。
    • 线性地址空间包含了系统中所有的段和系统表。
  • 逻辑地址转换为线性地址
    • 使用段选择器访问GDT或者LDT中相应的段描述符。
    • 检查段描述符中的访问权限和段长度限制。
    • 把段描述符中段的基地址和逻辑地址中的偏移量相加得到线性地址。
      ![逻辑地址和线性地址的转换](_v_images/20210305085803088_9859.png =715x)
  • 注意
    • 如果没有分页，则系统直接把线性地址当作物理地址。
    • IA-32e模式下的逻辑地址到物理地址的转换过程同上述过程。在64位模式下，逻辑地址的偏移量和段基址是64位而不是32位。线性地址格式也是64位宽，并且受规范格式要求的约束。
    • 如果进行了分页，则需要再将线性地址转换为物理地址。
