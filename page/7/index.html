OpenFlow协议解析

报文格式

Packet-In报文
Packet-Out报文

  • 由控制器向交换机发出的信息，用于将数据包发送到交换机的指定的端口。收到信息之后，交换机根据actions进行报文的处理。

  • 报文格式

    字段名称 描述
    version(8) OpenFlow协议版本
    OFPT_PACKET_OUT(8) 报文类型
    length(16) 报文长度
    Xid(32)
    Buffer_id(32) 交换机中存储数据包的缓冲区ID
    in_port(16) 交换机接收此数据包的端口或者为OFPP_CONTROLLER
    actions_len(16) action列表的长度，单位B
    actions_list 用于指导交换机对于数据包的转发
    数据(可选) 如果没有缓冲区ID则必须将报文传送回交换机

Ryu常用API

1. Ryu应用程序API

  • Ryu应用编程模型

    • Ryu应用是Ryu中实现各种功能的单线程实体。
    • Ryu事件是Ryu应用之间异步传递的信息，也可能来自Openflow控制器。事件可以包含任意的Python对象。
    • 每个Ryu应用维护一个事件的接收队列，采用FIFO调度方式。

  • 创建Ryu应用

    • Ryu应用是ryu.base.app_manager.RyuApp类的子类。
    • Ryu应用只支持单独的实例。

  • 监听事件

    • Ryu应用使用ryu.controller.handler.set_ev_cls修饰器监听特定的事件。

    • ryu.controller.handler.set_ev_cls(ev_cls,dispatcher=None)

      • ev_cls是一个事件类的实例，表示希望接收的事件,也可以为事件类实例的列表。

      • dispatcher说明函数被调用时交换机应该具有的状态。也可以是列表。

        交换机状态 描述
        ryu.controller.handler.HANDSHARK_DISPATCHER 发送并且等待hello信息
        ryu.controller.handler.CONFIG_DISPATCHER 已经进行版本协商和功能请求
        ryu.controller.handler.MAIN_DISPATCHER 接收到交换机属性信息，并且发送配置属性信息
        ryu.controller.handler.DEAD_DISPATCHER 断开连接

  • 生成事件

    • Ryu应用可以使用ryu.base.app_manager.RyuApp类中的方法生成事件。

  • 事件类

    • openflow事件类

      • openflow事件类继承于ryu.controller.ofo_event模块中的ryu.controller.ofp_event.EventOFPMsgBase(msg)类。用于描述接收到的由相连的交换机发出的Openflow信息。

      • openflow事件类至少包含以下属性

        属性 描述
        msg 用于描述相应的openflow信息的对象
        msg.datapath 用于描述openflow信息来源的交换机的对象，是ryu.controller.controller.Datapath的实例
        timestamp 信息源交换机生成Datapath实例的时间戳
        msg.reason

2. Ryu controller API

  • 用于实现控制器和交换机之间的互联和事件处理
2.1 controller API

  • 用于定义OpenFlow的控制器，用于处理交换机和控制器的连接等事件，同时还可以产生事件和路由事件

  • ryu.controller.controller.Datapath(socket,address)

    • 用于描述和本控制器相连的交换机

    • 实例拥有如下属性

      属性 描述
      id 64位的openflow Datapath ID，仅适用于ryu.controller.handler.MAIN_DISPATCHER状态
      ofproto 为相应的openflow版本给出openflow数据结构的定义的模块
      ofproto_parser 为相应的openflow版本给出openflow wire消息编码和解码器的模块
      ofproto_parser.OFPxxxx(datapath,…) 为给定的交换机生成openflow消息。可以使用Datapath.send_msg方法发送。xxxx是消息名，参数取决于消息。
      set_xid(self,msg) 生成一个openflow XID并且放到msg.xid中
      send_msg(self,msg) 对OpenFlow消息进行排队以发送到相应的交换机。如果msg.xid为None，则在排队之前会自动对该消息调用set_xid。
      send_barrier 将OpenFlow barrier消息排队发送到交换机

1. Verilog语法基础

1.1. 概述

  • **Verilog HDL**：是一种用于数字系统设计的语言。用此语言描述的电路设计即为该电路的Verilog HDL模型，也称为模块。
  • 特点：
    • 既是行为描述语言也是结构描述语言。即可用于建立描述电路功能行为的模块的模型也可用于建立描述元器件或者较大部件互联的模块的模型。
    • 可以是实际电路不同级别的抽象。
    • 一个复杂电路系统的完整Verilog HDL模型是由若干个Verilog HDL模块构成的，每一个模块又可以由若干个子模块构成。
    • 作为行为描述语言，语法结构适合与算法级和RTL级的模型设计；作为结构描述语言，适合门级和开关级的模型设计。
  • 抽象级别以及模型类型
    • 系统级(system-level)：用语言提供的高级结构能够实现待设计模块的外部性能的模型。属于行为描述
    • 算法级(algorithm-level)：用语言提供的高级结构能够实现算法运行的模型。属于行为描述
    • RTL级(register transfer level)：描述数据在寄存器之间的流动和如何处理、控制这些数据流动的模型。属于行为描述，与逻辑电路有着明确的对应关系。
    • 门级(gate-level)：描述逻辑门与逻辑门之间连接的模型。与逻辑电路有着确定的连接关系。属于结构描述
    • 开关级(switch-level)：描述器件中三级管和储存结点以及它们之间连接关系的模型。与物理电路有着明确的对应关系。属于结构描述

1. Verilog模块

1.1. 模块的结构

  • 是Verilog HDL的基本设计单元
  • 一个模块由两个部分组成：一部分描述接口，另一部分描述逻辑功能。即定义输入如何影响输出。

  • 引用模块时端口的连接
    • 严格按照模块定义时的端口顺序来连接，不用标明原模块定义时规定的端口名

    • 说明：类似于类的实例化为对象,如果实例引用时某个端口不需要，则只需要保留其逗号，无需写对应端口。

输出信息
  • 语法格式
    • 这两个函数和系统任务的作用都是用于输出信息，$isaplsy会在输出之后进行换行,$write则不会
    • 二者会将p2,pn按照p1给定的格式输出，p1称为格式控制
  • 输出格式控制
    • 是使用双引号引起来的字符串
    • 包含格式说明和普通字符
    • 格式说明
      • 以%开头
    • 普通字符
      • 原样输出的字符，特殊字符可以通过转换序列输出
    • 输出数据的显示宽度
      • 输出列表中数据的显示宽度总是为表达式的最大可能值的数据宽度，十进制时，结果前的0会被省略，但是其他进制不会省略
      • 举例
    • 不确定值或者高阻值的输出规则
      • 十进制

        所有位都为不定值，输出x
        所有位都为高阻值，输出z
        部分位为不定值，输出X
        部分位为高阻值，输出Z

      • 十六进制和八进制

        某一位数所对应的所有二进制位都为不定值，则该位输出x
        某一位数所对应的所有二进制位都为高阻值，则该位输出z
        某一位数所对应的部分二进制位为不定值，则该位输出X
        某一位数所对应的部分二进制位为高阻值，则该位输出Z

      • 对$fopen的每一次调用依次打开一个新的通道

    • 此语句总是在同时刻其他赋值语句执行完之后再执行，确保在同一时钟沿赋值的其他语句在执行完毕之后才显示数据。$display任务则是随机的
  • 举例
    值变转储文件
  • 值变转储文件(VCD)是一个ASCII文件，包含仿真时间、范围与信号的定义以及仿真运行过程中信号值的变化等信息。
  • 设计中的所有信号或者选定的信号集合在仿真过程中都可以被写入VCD文件，后处理工具可以把VCD文件作为输入并把层次信息、信号值和信号波形显示出来。
  • 命令
    • $dumpvars：选择需要转储的模块实例或者模块实例信号，$dumpvars(n,name)，n表示层次等级，即为只转储name所指模块的下第n层信号，n=0表示转储模块下各个层的所有信号
    • $dumpfile：选择VCD文件名称，$dumpfile("a.vcd")
    • $dumpon：选择转储过程的起点，启动转储过程
    • $dumpoff：选择转储过程的终点，停止转储过程
    • $dumpall：选择生成检测点

      $monitor
  • 作用：提供了监控和输出参数列表中的表达式或者变量值的可能
  • 语法格式
    • $monitor(p1,p2,……,pn);
      • 此时规则和$display一样。当参数列表中的变量或者表达式的值发生变化的时候，整个参数列表中的变量或者表达式的值都将显示，如果同一时刻多个参数值发生变化，则只显示一次
      • 参数可以是$time系统函数，此时参数列表中的变量或者表达式的值同时发生变化的时刻可以通过标明同一时刻的多行输出来显示。$monitor($time,,"rxd = %b txd = %b",rxd,txd);,,,表示空参数，输出为空格
      • $display中也可以使用上述用法
    • $monitor;
    • $monitoron;：用于控制$monitor的启动和停止，此命令用于打开监控标志，启动监控任务，调用次任务时总是会输出当前参数列表的值，用于设置初始值
    • $monitoroff;：同上，用于关闭监控标志，停止监控任务
  • 注意
    • 任何时刻只能由一个$monitor起作用，并且常在initial中调用

  • $time
    • 返回一个64位的整数表示当前仿真时刻值，该时刻是以模块的仿真时间尺度为基准。

    • 说明

1. Verilog语句

1.1. 赋值语句

1.1.1. 非阻塞赋值方式

  • 语法：a<=b;
  • 在语句块中，上面的语句所赋的变量不能立即为下面的语句所用。
  • 块结束后才能完成这次赋值操作，而所赋的变量值是上一次赋值得到的。
  • 在编写可综合的时序逻辑模块时，这是最常用的赋值方法。
  • 举例

  • 语法：b=a;
  • 赋值语句执行完后，块才结束。
  • 赋值语句结束之后变量的值立即改变。
  • 在时序逻辑中使用时，可能会产生意想不到的结果。

1.4. 块语句

1.4.1. 顺序块

  • 也称为过程块，由关键字begin-end组成
  • 特点
    • 语句顺序执行（除了带有内嵌延迟控制的非阻塞赋值语句）
    • 如果语句包含延迟或者事件控制，那么延迟总是相对于前面那条语句执行完成的仿真时间的。

      1.4.2. 并行块

  • 由关键字fork-join声明
  • 特点
    • 语句并发执行
    • 语句执行顺序由各自语句内延迟或者事件控制决定
    • 语句中的事件控制或者延迟是相对于块语句开始执行的时刻而言的
  • 注意
    • 如果两条语句在同一时刻对同一变量产生影响，那么会引起隐含的竞争，需要尽量避免

      1.4.3. 块语句的特点

  • 嵌套块
    • 块可以嵌套使用，顺序块和并行块可以混合使用
  • 命名块
    • 快可以拥有自己的名字，称为命名块
    • 语法格式
    • 特点
      • 命名块中可以声明局部变量
      • 命名块是设计层次的一部分，命名块中声明的变量可以通过层次名引用进行访问
      • 命名块可以被禁用

        1.4.4. 命名块的禁用

  • 通过关键字disable终止命名块执行，还可用于从循环中退出、处理错误条件以及根据控制信号控制某些代码段是否被执行
  • 类似于带标签的break

1.5. 生成语句

1.5.1. 概述

  • 可以动态的生成Verilog代码，方便了参数化模块的生成
  • 作用
    • 控制变量的声明、任务或者函数的调用，还能对实例引用进行全面的控制
  • 应用场景
    • 对矢量中的多个位进行重复操作
    • 进行多个模块的实例引用的重复操作
    • 根据参数的定义确定程序中是否应该包括某段Verilog代码
  • 生成的实例的类型
    • 模块
    • 用户定义原语
    • 门级原语
    • 连续赋值语句
    • initial和always块
  • 可以在生成范围内声明的数据类型
    • net(线网)，reg(寄存器)
    • integer(整型数)，real(实型数)，time(时间型)，realtime(实数时间型)
    • event(事件)
  • 特点
    • 生成的声明和生成的实例能够在设计中被多次的有条件的调用。
    • 生成的实例具有唯一的标识名，可以使用层次命名规则引用。
    • 生成的数据类型具有唯一的标识名，可以被层次引用
    • 任务和函数的声明也允许出现在生成范围之中，但是不能出现在循环生成当中，具有唯一的标识名，可以层次引用
  • 注意
    • 编写代码时必须在模块中说明生成的实例范围，使用关键字generate-endgenerate指定范围
  • 三种创建生成语句的方法

    • 条件生成
    • case生成

      1.6.3. task说明语句

  • task说明语句用于定义任务，和函数一起使用可以把一个很大的程序模块分解为许多较小的任务和函数便于理解和调试。

  • 特性

    • 任务完成之后，控制会传回启动过程，如果任务内部有定时控制，则启动的时间可以与控制返回的时间不同。
    • 任务可以启动其他任务，其他任务又可以启动别的任务，可以启动的任务数没有限制。
    • 只有当所有的启动任务完成之后控制才能返回。
    • 输入输出和总线信号的值可以传入、传出任务和函数。

  • 语法
